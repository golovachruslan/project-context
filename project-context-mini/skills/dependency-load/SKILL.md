# Load project-context-mini dependencies

Fetch the registered dependencies' `.project-context/` content (or single docs) into `.project-context/dependencies/<name>/`. Re-running is safe — each dep's subdirectory is wiped and re-populated.

**Clarify before acting:** if a fetch fails ambiguously (auth required, ref not found, redirect chain, target has no `.project-context/`), surface the error and use `AskUserQuestion` to ask the user how to proceed (skip / retry with different ref / mark broken). Prefer one batched question over guessing or asking one-at-a-time. Do not fabricate intent.

## Arguments

- `<name>` (positional, optional) — load only the dep with this name. Default: load all entries.
- `--clean` (optional flag) — wipe `.project-context/dependencies/<name>/` before fetching (default behavior anyway; the flag is informational).

## Workflow

### Step 1 — Read the manifest

```bash
cat .project-context/dependencies.json 2>/dev/null
```

If missing or empty, exit with:

> No dependencies declared. Run `/project-context-mini:dependency:add <source>` first.

If JSON is malformed, surface the parse error and exit. Do not attempt repair.

### Step 2 — Resolve target set

- If `<name>` was provided: filter the manifest to that one entry. Error if not found and exit.
- Otherwise: load all entries.

### Step 3 — Prepare the cache directory

```bash
mkdir -p .project-context/dependencies
```

Write/overwrite `.project-context/dependencies/.gitignore`:

```
# Auto-generated by project-context-mini:dependency:load
# Fetched dependency contexts are treated as a cache.
*
!.gitignore
```

### Step 4 — Dispatch per dep by `type`

For each target entry, wipe its subdirectory first:

```bash
rm -rf .project-context/dependencies/<name>
mkdir -p .project-context/dependencies/<name>
```

Then dispatch:

#### Type `git`

```bash
tmp=$(mktemp -d)
git clone --depth 1 --filter=blob:none --sparse --branch <ref> <source> "$tmp" 2>&1
git -C "$tmp" sparse-checkout set .project-context

if [ -d "$tmp/.project-context" ]; then
  # Copy only top-level .md files; skip subdirs like plans/, refs/
  find "$tmp/.project-context" -maxdepth 1 -type f -name '*.md' -exec cp {} ".project-context/dependencies/<name>/" \;
  result=ok
else
  result=no_context
fi

rm -rf "$tmp"
```

Note: use the `<ref>` field from the entry; default to `main` if missing.

If the clone itself fails (auth, network, ref not found): record `error` with the stderr message, do not crash, continue with the next dep.

#### Type `path`

```bash
src="<source>"   # resolved relative to repo root (parent of .project-context/)
abs=$(cd "$src" 2>/dev/null && pwd)

if [ -z "$abs" ] || [ ! -d "$abs" ]; then
  result=error  # path does not exist
elif [ -d "$abs/.project-context" ]; then
  find "$abs/.project-context" -maxdepth 1 -type f -name '*.md' -exec cp {} ".project-context/dependencies/<name>/" \;
  result=ok
else
  result=no_context
fi
```

#### Type `url`

Use `WebFetch` to retrieve the URL. Save the response body to `.project-context/dependencies/<name>/<basename>` where `<basename>` is the last path segment of the URL.

If the response is non-2xx or `WebFetch` errors, mark `error` with the status / message.

### Step 5 — Write per-dep metadata

For each dep that processed (even with `no_context` or `error`), write `.project-context/dependencies/<name>/.fetch-meta.json`:

```json
{
  "name": "<name>",
  "type": "<type>",
  "source": "<source>",
  "ref": "<ref-if-git>",
  "fetched_at": "<ISO 8601 UTC>",
  "result": "ok" | "no_context" | "error",
  "error": "<message-if-error>",
  "files": ["status.md", "architecture.md", ...]
}
```

Use `date -u +%Y-%m-%dT%H:%M:%SZ` for the timestamp.

### Step 6 — Summary

Emit a terse, grouped summary:

```
Loaded 3 of 4 dependencies into .project-context/dependencies/:
  ok       shared-types (git)     — 4 files
  ok       sibling-app (path)     — 3 files
  ok       rfc-001 (url)          — 1 file
  no_context  legacy-svc (git)    — remote has no .project-context/
```

If anything pulled non-trivially new architecture/flow content, suggest:

> Consider `/project-context-mini:update --chat` if any of these change how this project should be described.

## Edge cases

- **Empty manifest** — exit cleanly with the bootstrap hint, do not create the `dependencies/` directory.
- **Manifest has entries but `dependencies/` already contains stale subdirs from removed entries** — mention them in the summary as "stale" but do not auto-delete. User can `rm -rf` themselves.
- **Git ref not found** — record `error` with the git stderr message; user can fix the `ref` in `dependencies.json` and re-run.
- **Sparse-checkout returns no `.project-context/`** — the directory under `$tmp` will not exist; the script branches to `no_context`. This is the most common "broken" case for git deps and should be reported clearly.
- **URL redirects to a different file** — accept the final body. Use the original URL's basename for the saved filename to keep names predictable.
- **Source is a `path` to a project that uses the full `project-context` plugin** — its `.project-context/` likely contains `brief.md`, `state.md`, etc. We copy all top-level `.md` files regardless; mini does not enforce mini's filenames on dependencies.
- **`name` collision in manifest** — should not happen (`add` enforces uniqueness), but if it does, the second entry overwrites the first under the same subdir. Note in summary.

## What this skill does NOT do

- No `.git/` retention, no full-repo clones (sparse-checkout only).
- No diff against previous fetches. Each load is a clean wipe + re-copy.
- No CLAUDE.md sync, no commit prompt. Cache is gitignored.
- No automatic `update` after load — surfacing dep changes back into the four mini files is a separate, deliberate step.
- No reciprocal "we depend on you" notification to the source project.
