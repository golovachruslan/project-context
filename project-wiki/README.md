# project-wiki

An Obsidian-friendly, multi-project **LLM knowledge wiki** for Claude Code. Provide raw information from anywhere — Slack threads, emails, PDFs, design docs, loose notes — and it gets routed to the right project, archived immutably, and **compiled into a clean, cross-linked markdown wiki** that both engineers and AI agents can navigate fast.

It implements [Andrej Karpathy's LLM-wiki pattern](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f): three layers (immutable `raw/` → LLM-compiled `wiki/` → a schema doc), **compilation over retrieval** (a maintained index instead of a vector DB), `[[wikilinks]]`, and ingest / ask / lint operations.

Companion to [`project-context`](../project-context): that plugin manages *coding-session* context inside a code repo; **project-wiki manages a durable knowledge vault** (its own repo) across many projects.

## When to use

- You lead **multiple projects** and want one organized place for each project's technical details, POCs, notes, progress, and external references (repos, Slack channels, Google Docs).
- You want to **drop in raw info and have it filed correctly** — "here's a Slack export / this PDF / these notes" → it lands in the right project, structured.
- You want **AI agents and engineers to find related info efficiently**, with a structure that's explicit enough for an agent to navigate.
- You want **dependencies between projects** tracked, the Obsidian-native way.

## How it's organized

The plugin operates on a **vault** (a separate git repo / Obsidian vault). Point it at one via `--vault <path>`, the `PROJECT_WIKI_VAULT` env var, or a `.project-wiki` marker file.

```
<vault>/
├── CLAUDE.md / AGENTS.md   # schema block (managed section) — tells every agent how the vault works
├── index.md                # global catalog: every project + one-line summary + [[link]]
├── log.md                  # append-only operation log
├── dependencies.md         # cross-project graph (mermaid + edges), regenerated by lint
└── projects/<slug>/
    ├── _project.md         # hub: summary + status + key [[links]] + dependencies  (≤ 60 lines)
    ├── index.md            # catalog of this project's wiki pages, by type
    ├── progress.md         # current state only  (≤ 80 lines; history → progress/archive.md)
    ├── refs.md             # external refs: repos, Slack channels, docs (router signals)
    ├── raw/                # IMMUTABLE sources, date-prefixed (any format → extracted to text)
    └── wiki/               # COMPILED pages in typed subfolders, [[wikilinked]]
        ├── sources/  concepts/  entities/  poc/  decisions/  notes/
```

Every wiki page carries frontmatter — including a **mandatory `sources:`** list with evidence quotes — so every claim traces back to a raw source. This is the main guard against drifted or fabricated content.

## Install

> Published from the same repo as `project-context`. Until it's on a public marketplace, install from the local manifest (`.claude-plugin/marketplace.json`).

```
/plugin install project-wiki
/reload-plugins
```

You should then see `/project-wiki:init`, `:ingest`, `:ask`, and `:lint`.

## Skills

### `/project-wiki:init [--project <name>]`
Scaffold a vault, or add a project. Writes the layout above and inserts a managed schema block into `CLAUDE.md` / `AGENTS.md` **without clobbering** your existing content. `--project` asks five seed questions (name, repo, Slack channels, docs, dependencies) and wires dependency `[[links]]`.

### `/project-wiki:ingest [content|path] [--source ...] [--project <slug>]`
The heart. Source-agnostic (Slack / email / PDF / doc / note / paste). Routes the source to a project (asks if unsure), archives it immutably to `raw/`, then runs a two-pass **extractor → critic** pipeline to compile durable, cited wiki pages — you approve before anything is written. Enforces size budgets and logs the operation.

### `/project-wiki:ask [question] [--project <slug>]`
Answers from the compiled wiki with `[[page]]` citations. **Index-first** navigation, with a BM25 search fallback (`wiki_search.py`) for large projects. Optionally files a valuable answer back as a new page.

### `/project-wiki:lint [--fix] [--stats] [--project <slug>]`
Self-maintenance. Finds broken links, orphans, pages missing `sources:`, over-budget files, uncompiled raw, and shard thresholds. `--fix` relocates over-budget content, prunes dead links, shards oversized indexes, and regenerates `dependencies.md` — all with approval.

## Agents

- **`router`** — classifies which project a raw source belongs to (reads summaries + `refs.md`).
- **`wiki-extractor`** — over-generates candidate pages from one raw source (with source quotes).
- **`wiki-critic`** — ruthlessly filters candidates; prefers updating over creating; enforces budgets.
- **`wiki-explorer`** — navigates index-first + BM25 to answer questions with citations.

## Scripts (`scripts/`, pure stdlib Python 3.10+)

- **`wiki_lint.py`** — tiered health check (broken links, orphans, missing `sources:`, budgets, shard thresholds).
- **`wiki_search.py`** — frontmatter-filtered BM25 search (`--type` / `--tag` / `--since`). No embeddings, no vector DB.
- **`wiki_stats.py`** — page/link counts, density, orphans, which scaling threshold is approaching.

Run any of them standalone: `python3 scripts/wiki_lint.py <vault>`.

## Design choices

- **Compilation over retrieval** — a maintained index + BM25 fallback, deliberately *not* a vector DB. Right for ~handful of projects and hundreds of pages.
- **Obsidian-native dependencies** — `[[wikilinks]]` + a `dependencies.md` graph; Obsidian's graph view is the relational layer. No graph database to keep in sync.
- **Hard size budgets, relocate-never-delete** — hot files stay scannable; overflow moves to an archive/wiki page with a `[[link]]`. `raw/` keeps every source verbatim.
- **Mandatory source citations + surgical, re-read-the-raw updates** — guards against the LLM-wiki failure modes (silent corruption, output drift).
