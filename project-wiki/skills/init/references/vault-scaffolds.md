# Vault scaffolds

Templates the `init` skill writes. Vault-level files are created once; project files are created per `init --project`. Keep scaffolds short — the user fills `<TODO>` markers, and `ingest` deepens them over time.

The schema block below is the **single source of truth** for how the vault is structured. It is written into the vault's `CLAUDE.md` / `AGENTS.md` inside managed markers so every agent that opens the vault shares the same rules.

---

## Managed block for `CLAUDE.md`

Insert (or refresh) exactly this block between the markers. Never touch content outside the markers.

```markdown
<!-- PROJECT-WIKI:START -->
## Project Wiki — how this vault works

This is an LLM-maintained knowledge wiki (Karpathy LLM-wiki pattern). **Compile, don't re-retrieve:** answer from the maintained wiki, not by re-reading every raw source.

### Three layers
1. `projects/<slug>/raw/` — **immutable** source material (Slack, email, PDFs, docs), one date-prefixed file per source. Never edit these.
2. `projects/<slug>/wiki/` — **compiled** knowledge pages, in typed subfolders, cross-linked with `[[wikilinks]]`. The LLM owns this layer.
3. This schema — the operating rules.

### Layout
- `index.md` (global) — every project, one-line summary, `[[link]]`.
- `dependencies.md` — cross-project relationships (mermaid graph + list).
- `log.md` — append-only op log (ingest/query/lint), newest last.
- `projects/<slug>/`: `_project.md` (hub), `index.md` (page catalog), `progress.md` (current state), `refs.md` (external refs), `raw/`, `wiki/{sources,concepts,entities,poc,decisions,notes}/`.

### Finding things
Start at `index.md` → open the project's `_project.md` → use `index.md`/`indexes/` one-line summaries to pick pages → read only those. For large projects use `wiki_search.py` (BM25). Open `raw/` only to verify provenance.

### Page rules
- Every wiki page has frontmatter: `project`, `type` (`source|concept|entity|poc|decision|note`), `tags`, `updated` (YYYY-MM-DD), `status`, and **mandatory `sources:`** — a list of `{raw: [[...]], quote: "..."}`. Every non-trivial claim traces to a raw source.
- Link related pages with `[[wikilinks]]`. One concept per page.
- On update, **re-read the cited raw source** and edit the relevant section surgically — never blind-rewrite.

### Size budgets (hard — relocate, never delete; raw/ keeps everything)
- `_project.md` ≤ 60 lines (hub: summary + status + links). Overflow → a wiki page + `[[link]]`.
- `progress.md` ≤ 80 lines (current state only). Old completed items → `progress/archive.md` + summary line.
- wiki page: soft 400 / hard 800 lines → split into `<type>/<topic>/refs/*.md`.
- `index.md` > ~300 lines or > ~150 pages → shard into `indexes/<type>.md`.

### Operations
`/project-wiki:ingest` (raw in → routed → compiled), `/project-wiki:ask` (query, cited answer), `/project-wiki:lint` (health + budgets), `/project-wiki:init --project <name>` (add a project).
<!-- PROJECT-WIKI:END -->
```

## Managed block for `AGENTS.md`

Shorter pointer (same markers):

```markdown
<!-- PROJECT-WIKI:START -->
## Project Wiki

LLM-maintained knowledge wiki. Read `CLAUDE.md` for the full schema. In short:
- `projects/<slug>/raw/` = immutable sources; `projects/<slug>/wiki/` = compiled, `[[wikilinked]]` pages.
- Find info: `index.md` → project `_project.md` → page catalog → read only the pages you need; `wiki_search.py` for scale.
- Every wiki page cites a raw source (`sources:` frontmatter). Compile, don't re-retrieve.
<!-- PROJECT-WIKI:END -->
```

### `_templates/wiki-page.md`
```markdown
---
project: <slug>
type: concept
tags: []
updated: <YYYY-MM-DD>
status: draft
sources:
  - raw: "[[<raw-file-stem>]]"
    quote: "<verbatim evidence from the source>"
---

# <Page Title>

<body — one concept, cross-linked with [[wikilinks]]>
```

### `_templates/raw-source.md`
```markdown
---
source: <slack|email|pdf|doc|note|paste>
origin: "<channel | url | sender>"
date: <YYYY-MM-DD>
project: <slug>
---

<verbatim source content — never edited after archiving>
```

## Per-project files (written by `init --project`)

### `_project.md` — from `_templates/project.md`, prefilled from seed answers.

### `index.md`
```markdown
# <Project Name> — Index

One-line summaries of every wiki page, grouped by type. Pick a page from here, then read it.

## Concepts
## Entities
## POCs
## Decisions
## Notes
## Sources
```

### `progress.md`
```markdown
# Progress — <Project Name>

## Current Focus
<TODO: one sentence on what's active now>

## In Progress
- <TODO>

## Recent
- <YYYY-MM-DD> <TODO: recently completed>
```
Keep ≤ 80 lines. Older completed items move to `progress/archive.md`.

### `refs.md`
```markdown
# References — <Project Name>

External pointers the router and humans use to locate this project.

- **Source code:** <repo URL(s)>
- **Slack:** <#channel(s)>
- **Docs:** <Google Doc / Notion / Confluence links>
- **Dashboards / other:** <links>
```

## Seed questions for `init --project`

Ask these as one batch:

1. **Name & summary** — what is this project and who uses it?
2. **Source code** — repo URL(s).
3. **Slack** — channel(s) the project lives in (these become routing signals).
4. **Docs** — Google Docs / Notion / other reference links.
5. **Dependencies** — which existing projects does it depend on (upstream) or feed (downstream)?

Accept brief answers. `ingest` fills in the wiki over time.
